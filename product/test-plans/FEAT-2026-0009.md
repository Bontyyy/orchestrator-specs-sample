## Coverage notes

Five tests covering three acceptance criteria. Default `qa_authoring` cardinality applied — the feature spec's `## Scope` section does not declare a cardinality override.

**AC-1 coverage (2 tests):** `widgets-bulk-create-single` and `widgets-bulk-create-multiple`. The single-item test validates the minimum batch size and confirms the response shape (array with generated `id`). The three-item test validates a multi-item batch including boundary `quantity` values (0 and 10000) to confirm valid edge cases are accepted without error. Both tests assert HTTP 201 and the presence of a generated `id` on each created object.

**AC-2 coverage (1 test):** `widgets-bulk-create-batch-size-exceeded` sends a 51-item batch (one over the limit) generated at test time via `python3`, to avoid maintaining a large inline JSON string. Asserts HTTP 400, `error.code == "batch_size_exceeded"`, and a non-empty `error.message` referencing the 50-item limit.

**AC-3 coverage (2 tests):** `widgets-bulk-create-validation-failure-response` covers the response shape — HTTP 422, `error.code == "batch_validation_failure"`, `error.failures` array with `index` and `reason` per entry. The batch contains one valid item (index 0), one item with blank `name` (index 1), and one item with blank `sku` (index 2), exercising two distinct validation rules. `widgets-bulk-create-atomicity` is the CRITICAL atomicity test: it captures `GET /widgets/count` before and after a mixed batch POST and asserts the count is unchanged, verifying that the valid item at index 0 was NOT persisted alongside the invalid item at index 1. The python3 assertion provides a clear, executable pass/fail predicate.

**Runtime port: `5083`**, sourced from `src/OrchestratorApiSample.Api/Properties/launchSettings.json`, profile `http` (`applicationUrl: "http://localhost:5083"`). The first test's `commands[0]` backgrounds the service startup with `&` and `commands[1]` pauses 3 seconds for port binding; subsequent tests assume the service is running. qa-execution must confirm port 5083 is bound before evaluating test 1's `expected` predicate.

**Atomicity test note:** The atomicity test (`widgets-bulk-create-atomicity`) relies on `GET /widgets/count` returning a JSON object with a `count` integer field. This endpoint exists in the component repo (`WidgetsController.cs` line 75, `[HttpGet("count")]`, returns `Ok(new { count })`). The test uses the in-memory store's isolated state; if qa-execution runs tests in parallel or reuses a shared store, the before/after count comparison may be affected by concurrent widget creation — qa-execution should run these tests sequentially in the declared order.
